## 9. 专家分析回复（2026-02-27）

### 问题清单逐一回答

**Q1. 状态机设计：单一事实源应放在何处？**

推荐 **`HTMLAudioElement` 的实际播放状态（`audio.paused`、`audio.currentTime`）作为兼容模式下的事实源**，`audioPlaying` 仅作为”意图态”驱动指令发出。在 mpv 模式下，事实源切换到 mpv 进程的 `pause`/`time-pos` 属性。

当前代码的核心矛盾是：`audioPlaying`（UI 意图态）和 `audio.paused`（真实态）在模式切换窗口内脱钩后无人负责对账。正确做法是在切换完成后增加一次**强制对账（reconciliation）**，用真实态覆盖意图态。

**Q2. 切换事务：是否需要两阶段事务？**

不需要完整的 prepare/commit 协议，但需要一个 **”切换收敛确认”步骤**。当前缺少的关键环节是：转换完成后验证 `<audio>` 是否真正进入播放态。建议在切换 effect 末尾增加一个延迟对账（~300ms），用 `!audio.paused && audio.currentTime` 在变化来确认播放已生效。

**Q3. 事件治理：onPlay/onPause 与主动同步 effect 如何拆分？**

- `onPlay`/`onPause`：仅在 `audioEngineMode === 'chromium'` 且 `suppressNativePlaybackEventsRef.current === false` 时写回 `audioPlaying`。当前实现正确。
- 主动同步 effect（`MusicMainSection.tsx:545-568`）：应增加”模式切换中”的保护标志，避免在过渡窗口内与 transition effect 竞争操控 `<audio>` 元素。
- 推荐引入 `modeTransitionInProgressRef` 标志，在 transition effect 开始时设 `true`，在收敛确认回调中设 `false`。sync effect 在此标志为 `true` 时跳过执行。

**Q4. 时钟连续性：如何稳定继承播放时间？**

当前实现已具备基本的时钟继承逻辑（`MusicMainSection.tsx:468-475`，用 `audioTime` 作为 seek 目标）。由于 mpv 模式下 `<audio>` 同样在播放（静音），`audioTime` 由 `<audio>` 的 `onTimeUpdate` 驱动，两者偏差通常 < 0.6s，因此 seek 多数情况下不触发。

当前方案可行，但如果未来 mpv 与 `<audio>` 出现漂移（例如 mpv seek 后 `<audio>` 未跟随），建议 mpv 模式下禁止 `<audio>` 的 `onTimeUpdate` 写回 `audioTime`，改由 mpv 的 `time-pos` property-change 事件驱动 UI 时钟。这是 P1 改进项。

**Q5. 设备/上下文冲突：AudioContext 是否应解耦？**

**这正是当前 bug 的直接诱因**（详见下方根因树 A 节）。`createMediaElementSource()` 将 `<audio>` 的输出永久路由到 Web Audio API 图。当可视化运行时 dispose 关闭 AudioContext 后，`<audio>` 的输出管道断裂。切回兼容模式时新建的 AudioContext 默认为 `suspended` 状态，导致解码器阻塞。

短期建议（P0）：切换回兼容模式时主动 resume AudioContext。
中期建议（P1）：mpv 模式下不 dispose AudioAnalyser，仅暂停采样；或将 visualizer 分析通道与 `<audio>` 解耦，使用 OfflineAudioContext 或独立数据源。

**Q6. 最小可行修复顺序：优先落哪 2~3 个改动点？**

详见下方 B 节（P0 最小修复方案），3 个改动点：
1. mpv → chromium 切换时调用 `resumeAudioAnalyser()`
2. 增加延迟对账机制
3. transition effect 移除 `audioTime` 依赖

### B. 最小修复方案（P0）

#### 改动 1：mpv → chromium 切换时主动 resume AudioContext

**文件**：`src/components/MusicMainSection.tsx`
**位置**：transition effect（当前第 433-494 行），mpv → chromium 分支末尾

**改法**：在 `audio.play()` 调用后，追加 `resumeAudioAnalyser()` 调用，确保 AudioContext 进入 running 状态。

```typescript
// 在 if (previousMode === 'mpv' && audioEngineMode === 'chromium') 分支末尾
if (!audioPlaying) {
  audio.pause()
  return
}

void resumeAudioAnalyser()
void audio.play().catch(() => {
  setAudioPlaying(false)
})
```

**注意**：`resumeAudioAnalyser()` 中的 `attach()` 调用会在 AudioContext 尚未创建时新建一个。此处主动调用确保在 transition effect 的同步上下文中（用户手势传导链内）创建 AudioContext，使其进入 `running` 状态而非 `suspended`。这也解决了 RAF 延迟创建的时序问题。

但仍有一个风险：transition effect 运行在 React effect commit 阶段，此时已不在用户手势的调用栈内。Chrome 的 AudioContext autoplay policy 可能仍然让新建的 context 进入 suspended。因此需要改动 2 作为保底。

#### 改动 2：增加延迟对账机制

**文件**：`src/components/MusicMainSection.tsx`
**位置**：transition effect mpv → chromium 分支末尾

**改法**：在 play() 后调度一个 ~350ms 延迟的对账回调，确认 `<audio>` 真正进入播放态，否则重新 resume + play。

```typescript
// 在 transition effect 的 mpv → chromium 分支末尾追加
const reconcileTimer = window.setTimeout(() => {
  const currentAudio = audioRef.current
  if (!currentAudio || !focusedAudioSrc) {
    return
  }
  // 尝试 resume AudioContext（可能在 RAF 中被创建为 suspended）
  void resumeAudioAnalyser()
  // 对账：如果意图是播放但实际暂停/卡住，强制重试
  if (audioPlaying && currentAudio.paused) {
    void currentAudio.play().catch(() => {
      setAudioPlaying(false)
    })
  }
}, 350)
```

cleanup 中需清理此 timer（通过 ref 持有 timer id）。

#### 改动 3：transition effect 移除 `audioTime` 依赖

**文件**：`src/components/MusicMainSection.tsx`
**位置**：transition effect 的依赖数组

**改法**：`audioTime` 会随 `onTimeUpdate` 高频变化（~4Hz），导致 transition effect 反复重入（虽然 ref guard 使其成为 no-op，但仍浪费资源且增加竞态风险）。将 `audioTime` 从依赖数组中移除，改用 `audioRef.current.currentTime` 读取即时值。

```typescript
// 替换
const resumeAtSec = clamp(audioTime, 0, Math.max(0, audioDurationSec))

// 为
const currentAudioTime = audioRef.current?.currentTime ?? 0
const resumeAtSec = clamp(
  Number.isFinite(currentAudioTime) ? currentAudioTime : 0,
  0,
  Math.max(0, audioDurationSec),
)

// 依赖数组移除 audioTime：
// [audioDurationSec, audioEngineMode, audioMuted, audioPlaying, audioVolume,
//  focusedAudioSrc, suppressNativePlaybackEventsFor]
```

同理，`audioPlaying`、`audioMuted`、`audioVolume` 在 transition effect 中也可改为 ref 读取以避免不必要的重入，但这是可选优化。

### C. 结构化重构方案（P1）

#### C1. AudioAnalyser 生命周期与模式切换解耦

**目标**：mpv 模式下不关闭 AudioContext，仅暂停采样。

**改法**：

1. 在 `useMusicVisualizerRuntime.ts` 中，将 `active=false` 的处理从 `dispose()` 改为 `suspend()`：
   ```typescript
   if (!active) {
     audioAnalyserRef.current?.suspend() // 新增方法：暂停采样但保留 context
     // ...
     return
   }
   ```

2. 在 `MusicAudioAnalyser` 中新增 `suspend()` 方法：
   ```typescript
   suspend(): void {
     // 不 close context，不 disconnect nodes
     // 仅停止 sample() 调用（由 render loop 控制）
     if (this.audioContext && this.audioContext.state === 'running') {
       void this.audioContext.suspend()
     }
   }
   ```

3. 重新激活时调用 `resume()` 而非重新 `attach()`。由于 AudioContext 未关闭、MediaElementSourceNode 未断开，`<audio>` 的输出管道保持完整。

**收益**：彻底消除 AudioContext close/recreate 循环带来的管道断裂风险。切换模式变为”AudioContext suspend/resume”而非”destroy/create”。

#### C2. 引入 modeTransitionInProgressRef 保护

**目标**：消除 transition effect 与 sync effect 的竞态。

**改法**：

1. 新增 `const modeTransitionInProgressRef = useRef(false)`
2. transition effect 开始时设为 `true`，在延迟对账回调（或确认播放成功后）设为 `false`
3. sync effect（`MusicMainSection.tsx:545`）在 `modeTransitionInProgressRef.current === true` 时 `return` 跳过

#### C3. mpv 时钟驱动 UI（可选）

**目标**：mpv 模式下 UI 时钟由 mpv 的 `time-pos` 属性驱动，而非 `<audio>` 的 `onTimeUpdate`。

**改法**：

1. 在 `audioEngineController.ts` 中监听 mpv 的 `property-change` 事件（`time-pos`）
2. 通过 IPC 推送到 renderer
3. mpv 模式下禁止 `onTimeUpdate` 写回 `audioTime`
4. 兼容模式下恢复 `onTimeUpdate` 驱动

**收益**：消除 mpv 与 `<audio>` 时钟漂移；为后续”mpv 模式下 `<audio>` 完全不播放”铺路。

### E. 风险与回滚策略

#### P0 改动风险

| 改动 | 风险 | 缓解 |
|------|------|------|
| 改动 1：transition 中调用 resumeAudioAnalyser | AudioContext 可能在 effect 中创建（非用户手势），Chrome 仍给 suspended | 改动 2 的延迟对账兜底 |
| 改动 2：350ms 延迟对账 | 350ms 内用户可能已操作（如切歌），对账逻辑与新操作冲突 | 对账回调中检查 focusedAudioSrc 与 audioEngineMode 是否仍为预期值，不匹配则跳过 |
| 改动 3：移除 audioTime 依赖 | 使用 audioRef.current.currentTime 读取即时值可能与 React 渲染不一致 | transition 场景下只需要一个 “近似位置” 作为 seek 目标，误差 < 1s 可接受 |

#### P1 改动风险

| 改动 | 风险 | 缓解 |
|------|------|------|
| C1：AudioAnalyser suspend 替代 dispose | AudioContext.suspend() 后内存不释放，长时间不使用可视化时浪费资源 | 增加超时（如 5 分钟无可视化活动则 dispose），或在应用非 active 时 dispose |
| C2：modeTransitionInProgressRef | 如果对账回调未执行（如组件卸载），ref 永久为 true，sync effect 永久跳过 | cleanup 中重置 ref |
| C3：mpv 时钟驱动 | 新增 IPC 推送通道，增加主进程与渲染进程通信复杂度 | 使用节流推送（~4Hz），与现有 onTimeUpdate 频率对齐 |

#### 回滚策略

- P0 的 3 个改动均在 `MusicMainSection.tsx` 的 transition effect 中，可通过 revert 单个 commit 回滚。
- P1 改动涉及 `audioAnalyser.ts` 和 `useMusicVisualizerRuntime.ts`，建议独立 branch + PR。
- 所有改动不影响 mpv 主进程引擎层（`audioEngineController.ts`/`mpvEngine.ts`/`mpvHost.ts`），后端无回滚需求。

### F. 对第 4 节问题归纳的修正

原判断”不是引擎模式切换失败，而是 `audioPlaying` 与 `HTMLAudioElement` 再次失配”——**方向正确但未触及根因**。

实际根因不在 React 状态管理层，而在 **Web Audio API 的 AudioContext 生命周期管理层**：

1. `createMediaElementSource()` 将 `<audio>` 输出永久路由到 AudioContext 图
2. mpv 模式下 visualizer dispose 关闭了 AudioContext → `<audio>` 输出管道断裂
3. 切回 chromium 时新 AudioContext 以 `suspended` 创建 → `<audio>` 解码器被阻塞
4. `resumeAudioAnalyser()` 的触发条件（`audioPlaying` 变化）在此场景下不满足
5. 结果：`audioPlaying=true`（意图态）但 `audio.paused=false + currentTime 冻结`（真实态卡死）

用户 workaround “暂停→播放” 之所以有效，是因为 `toggleAudioPlayback()` 的 play 分支调用了 `resumeAudioAnalyser()`，后者执行 `audioContext.resume()` 解除了阻塞。
