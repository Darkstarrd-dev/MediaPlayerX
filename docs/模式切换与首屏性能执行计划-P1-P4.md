# 模式切换与首屏性能执行计划（P1~P4）

Last updated: 2026-02-22

## 0. 目标与执行纪律

- 目标问题：数据库规模上升后，`image/video/music` 模式切换和重启首屏加载延迟严重。
- 本计划按 **P1 -> P2 -> P3 -> P4** 严格分阶段执行，前一阶段未验收通过，不进入下一阶段。
- **强约束**：每阶段仅允许读取该阶段“允许读取文件”清单；禁止预读取与本阶段无关的内容。
- 每阶段均包含 TODO checklist、验收指标、回滚点。

### 4.14 P4 专家评审回复（2026-02-21）

- 评审前提
  - 4.13 的数据复核已证明：修正样本后 stage-2d 在主链路指标上**全面优于 P3 tuned**（`elapsed -37~-41%`、`e2e_ready -4~-5%`、`data_commit -5~-23%`、`resolve -9~-27%`、`readImportTasks -49~-54%`），仅 `longtask.total_ms` 有 `+2~+3%` 微幅上升。
  - 此前多轮"P4 存在明确回归"的判定，系样本噪声放大所致（如 2.8 的 ready 反向、3.7 的 OFF 组 ready 回归均属同类问题）。
  - **核心判断：P4 当前陷入的不是"优化方向错误"的瓶颈，而是"测量精度不足导致无法区分信号与噪声"的瓶颈。**

#### 路线 A（稳口径 Rebaseline）— 可行性：高 | 正确性：高 | 风险：低

- 评审结论：**必须最先执行**。
- 依据
  - 4.13 复核中，仅更换样本计算方式即可将 stage-2d 的结论从"回归"反转为"改善"。说明当前测量体系的置信度不足以支撑任何后续优化方向的判定。
  - `longtask.total_ms` 的 `+2~+3%` 变化完全在抖动噪声范围内（对比 4.8 中 OFF 组 `longtask` pair jitter 曾达 `90.26%`），不构成有效信号。
  - 没有可信基线，路线 B~E 的复测结果同样无法被信任。
- 执行要求
  - 同一 git commit、同一导入目录、同一初始选中包。
  - ON/OFF 交错各 `>=3` 轮。
  - 报告中位数 + pair jitter + IQR；单指标 pair jitter `>15%` 的维度需标注为"不可信"。

#### 路线 B（前端需求削峰）— 可行性：高 | 正确性：高 | 风险：低

- 评审结论：**推荐第 2 步执行**，ROI 最高的低成本变更。
- 依据（基于代码审查）
  - `resolveMediaResource` 调用链较重：`ensureSnapshotLoaded()` → `cleanupExpiredTokens()` → `assertLocatorAllowedWithRecovery()`（含 archive index refresh）→ `maybeResolveThumbnailLocator()`（含 CPU 密集的 image decode → resize → WebP encode）→ token 发放。
  - 在 `importBusy=true` 期间，前端可能为大量**尚未可见**的条目发起 resolve 请求。限制为仅解析当前可视区域 + 焦点邻域的缩略图，可直接减少 IPC 并发压力。
  - 间接收益：降低 `ensureSnapshotLoaded()` 的并发争抢概率、减少 archive index refresh 的连锁触发。
  - 纯前端变更，不涉及后端改造，可独立回滚。

#### 路线 C（resolve 快速路径）— 可行性：中 | 正确性：有条件正确 | 风险：中

- 评审结论：**不建议单独执行**，应与路线 B 合并考虑。
- 依据（基于代码审查）
  - **archive 场景限制**：对 `kind === 'archive-entry'` 的 locator，"原图"本身也需 ZIP 解压（`readArchiveEntryMedia`），并不比缩略图生成轻量太多。快速路径的收益被此场景显著削弱。
  - **二次加载闪烁**：先展示原图、缩略图就绪后替换会导致视觉跳动，需要额外 UI 状态管理（placeholder → full → thumbnail transition），增加前端复杂度。
  - **带宽反增**：直接返回原图意味着更大传输体积，在大量并发 resolve 时可能反向加重主线程序列化负担。
- 可行组合方式
  - 路线 B 先削减总请求量，路线 C 仅对**剩余的 filesystem locator**（非 archive）启用快速路径。

#### 路线 D（主进程 QoS 优先级调度）— 可行性：低-中 | 正确性：存在根本性限制 | 风险：中

- 评审结论：**不建议当前执行**。
- 依据（基于代码审查）
  - **根本性约束**：数据库层使用 `DatabaseSync`（`mediaLibraryDatabase.ts:15-16`，同步 SQLite API）。所有读写操作在主线程串行执行，不可被抢占。当导入任务执行 `replaceSnapshot()`（`BEGIN IMMEDIATE → COMMIT` 事务）时，整个主线程被阻塞；无论为交互读请求设置多高优先级，都必须等事务结束。
  - **QoS 仅控制"下一个读谁"，无法控制"当前正在阻塞的操作何时结束"**。在 `DatabaseSync` 同步性限制下，优先级排队只能起到边际效果。
  - **隐式 QoS 已存在**：当前 stage-2d 的 supersede 机制本质上实现了"最新交互请求优先"——过时请求被 abort、最新请求替代旧请求执行，效果等同于交互优先调度。
  - **数据不支持**：`readImportTasks` 在 stage-2d 中已表现良好（vs P3 `-49~-54%`），说明导入读取本身不是当前瓶颈。
- 前置条件
  - 仅在路线 E 完成后（读路径异步化），QoS 才能发挥实际效果。在 `DatabaseSync` 上做 QoS 是投入错误层级。

#### 路线 E（Worker/异步通道迁移）— 可行性：低（高工程量） | 正确性：方向正确 | 风险：高

- 评审结论：**方向正确但当前不应进入**，仅在路线 A+B 完成后指标仍不达标时再评估。
- 依据
  - 4.13 复核数据已显示 stage-2d 在主链路指标上全面优于 P3 tuned。当前的优化幅度可能"已经足够好"，路线 E 的边际收益不一定能覆盖重构成本。
  - **工程复杂度极高**：
    - `DatabaseSync` 不能跨线程共享（SQLite in-process 状态），Worker 需要独立的 `DatabaseSync` 实例。
    - `LibrarySnapshotService` 持有大量内存状态（`archiveEntryIndexByPath`、`zipEntryIndexByPath`、`normalizedArchiveCacheBySourcePath`），跨线程同步这些状态引入新的一致性管理复杂度。
    - 主线程 ↔ Worker 之间的 `structuredClone` 序列化成本可能抵消部分收益（尤其是 ~97MB 量级的快照数据）。
  - 如要启动，建议采用最小方案：**仅迁移读路径到 Worker，写路径保持主线程**，利用 WAL 模式允许并发读的特性。

#### 总评

| 路线 | 优先级 | 预期 ROI | 执行建议 |
|------|--------|----------|----------|
| A（稳口径） | **必须** | 高（消除误判） | 第 1 步 |
| B（前端削峰） | 高 | 高（低成本高回报） | 第 2 步 |
| C（resolve 快速路径） | 中低 | 中（受 archive 场景限制） | 与 B 合并，不单独做 |
| D（主进程 QoS） | **不建议** | 低（被 DatabaseSync 同步性根本限制） | 跳过 |
| E（Worker 迁移） | 后置 | 高但风险也高 | A+B 不足时再评估 |

- 对 4.13 推荐顺序的修正
  - 原方案：A → B+C → D → E。
  - 修正方案：**A → B（+C 仅 filesystem locator） → E（若仍不达标）**。
  - 路线 D 应**跳过**：`DatabaseSync` 同步性使其预期收益极为有限，投入产出比远不如 B。不应作为独立阶段消耗验证资源。
  - 路线 C 应**降级为 B 的子项**：archive 场景限制使其独立价值不足，不值得作为独立阶段管理。
- 核心认知
  - **P4 stage-2d 可能已经"足够好"。** 4.13 数据复核证实了这一点。P4 真正缺的不是更多优化方向，而是一个可信的测量体系来确认现有成果。在路线 A 完成之前，任何新增优化都存在被噪声误判的风险。

### 4.16 路线 A 执行期间异常观测与根因分析（专家评审补充，2026-02-21）

- 运行期间观测到的异常现象
  - **后端超时 error banner**（间歇出现）：
    - "数据快照: 后端请求超时（>8000ms）"
    - "Sidebar 目录树: 后端请求超时（>8000ms）"
    - "运行时依赖预检: 运行时能力探测失败：后端请求超时（>6000ms）"
  - **导入状态面板出现多条历史导入条目**（同时可见约 15 条）：
    - `文件夹 | 2842/2842 构建容器完成`
    - `文件夹 | 2737/2737 薄扫描进行中，已处理 223991 个文件`
    - `文件夹 | 1421/1421 薄扫描进行中，已处理 130600 个文件`
    - 以及 `807/456/790/187/302/672/290/837/120/44` 等多条
    - 用户确认"本轮执行的再完成后会清除"
  - **浏览完成后应用迟迟不退出**：UI 已加载完毕但 bench 继续运行很长时间
  - **系统资源特征**（任务管理器）：CPU 3-4%、内存 400MB→1.4GB 持续增长、磁盘吞吐接近 0、GPU 在 browse 阶段出现 ~10% 峰值（5090 级别）

#### 根因 1：多轮测试导入状态泄漏（路线 A 高抖动的首要原因）

- 代码审查结论
  - `E2eBenchController.tsx` 在 browse+import 模式启动时**不调用 `clearDatabase()`**（L362-409），依赖 `--library-root` 文件系统隔离。
  - 但交错复测使用**同一个 `library-root`**（统一导入路径 `Z:\Selection\002Errors`），前一轮的导入任务记录和快照缓存**在新进程启动时被恢复**。
  - `importTaskService.recoverInterruptedImportTasks()`（L38-59）将旧 'running'/'pending' 任务标记为 'failed'，但**不删除任务记录**，`readImportTasks()` 仍返回全部历史任务。
  - 每轮新增的 `enqueueImportTask()` 叠加在已有导入源上，导致 `librarySnapshotService` 的 refresh 需要重新扫描**所有已注册导入源**。
- 后果
  - 第 N 轮运行时，后台同时处理当前轮 + N-1 轮的累积导入源扫描。
  - 每轮的后台负载不同 → 主线程争抢不同 → 关键指标抖动放大。
  - 这是 4.15 中 pair jitter 全面 `>40%`（远超 15% 阈值）的**首要解释**。
  - 导入面板同时出现 ~15 条条目直接证实了此判断。

#### 根因 2：8000ms IPC 超时触发（主线程阻塞的直接证据）

- 超时阈值
  - `DEFAULT_IPC_TIMEOUT_MS = 8_000`（`useReadOnlyDataAccess.ts:24`）。
  - 运行时依赖预检超时阈值为 `6_000ms`。
- 触发原因
  - 2842 个容器 / 223,991 个文件的 snapshot 读取在 `DatabaseSync`（同步 SQLite）上构造全量对象图 → Zod `.parse()` 深拷贝 → JSON 序列化。
  - 单次 `readSnapshot()` 周期的堆分配峰值约 payload × 3~4（原始对象 + Zod clone + JSON string），对此规模数据 payload 约 97MB+，峰值堆压力约 300~400MB。
  - 叠加导入源累积（根因 1），后台 refresh 扫描持续消耗主线程时间，IPC 响应被延后到超过 8s。
- 与之前资源观测的一致性
  - CPU 3-4%（单线程受限于内存分配/GC）、磁盘接近 0（SQLite 已在 page cache）、内存持续增长（多层深拷贝累积）、GPU ~10% browse 阶段峰值（Chromium compositor 纹理上传）——全部指向**内存受限的工作负载**，而非 I/O 或 CPU 受限。

#### 根因 3：browse 完成后迟迟不退出

- `wait-import-completion=1` 使 bench 进入 `awaiting_import` 轮询循环（`E2eBenchController.tsx:413-464`），每 1.5s 调用 `readImportTasks({ timeoutMs: 8_000 })`。
- 当导入源累积时，后台 refresh 扫描更多文件夹，实际导入完成时间远超预期。
- 同时 `readImportTasks()` 的 IPC 响应本身也可能超时（根因 2），进一步拉长等待。

#### GPU ~10% 峰值评估

- 来源确认：非应用主动 GPU 计算。字幕 ASR（`subtitleSession.ts`）未激活、音乐可视化（`webglRenderer.ts`）未激活、缩略图渲染使用 Sharp CPU 编码。
- 实际来源为 **Chromium GPU 合成器**：browse 阶段快速翻页时，大量图片解码 → GPU 纹理上传 → 层合成。
- 结论：对 5090 级别显卡的 10% 占用属于 Chromium 渲染开销，在 bench 密集翻页场景下正常，正常用户操作频率下会显著低于此值。
- 与路线 B 的关联：减少同时 resolve 的缩略图数量可间接降低 GPU 合成器的纹理上传负载。

### 4.17 专家评审更新（综合 4.14-4.16，2026-02-21）

- 整体判断更新
  - 4.14 的评审方向仍然正确，但 4.15-4.16 的新证据揭示了一个**比"测量噪声"更根本的问题**：bench 基础设施本身存在缺陷（导入状态泄漏），且系统在 2842 容器规模下已经触及**内存受限瓶颈**（8s IPC 超时、400MB→1.4GB 堆增长）。
  - 原来认为"P4 stage-2d 可能已经足够好"的结论**需要附加条件**：对 `002Errors` 规模（~2800 容器、~97MB payload）可能足够好，但对用户实际场景（200GB、更大 payload）尚未验证且已观测到超时错误。

#### 路线 A 修正（bench 基础设施治理，前置于一切复测）

- **当前路线 A 失败不是因为"测量抖动本身无法控制"，而是因为 bench 存在状态泄漏 bug**。
- 必须修复后再重跑：
  - **修复 1**：`E2eBenchController` 在每轮启动时（`phase='warmup'` 之前），如果检测到已有导入源存在，应调用 `clearDatabase()` 或至少清理历史导入任务记录。确保每轮运行从相同的干净状态开始。
  - **修复 2**：`importTaskService.recoverInterruptedImportTasks()` 应增加删除已完成/已失败的旧导入任务记录的逻辑，避免 `readImportTasks()` 返回历史残留。
  - **修复 3**：交错复测脚本应为每轮运行使用独立的 `user-data-dir`，或在 seed 阶段后显式拷贝干净快照作为后续轮次的起始状态。
- 修复后预期：pair jitter 应大幅收敛，路线 A 有望通过。

#### 路线 F（新增：结构性内存治理）— 可行性：高 | 正确性：高 | 风险：低-中

- 评审结论：**应在路线 A（修复后）和路线 B 之间插入，或与路线 B 并行执行**。
- 依据（基于代码审查 + 运行期资源观测）
  - `readSnapshotLite()` 的实现（`mediaLibrarySnapshotStore.ts:563-566`）先调用 `readSnapshot({ includeImages: false })` 构造完整 DTO 对象图，然后**再过一次 `librarySnapshotLiteDtoSchema.parse()`**——即同一份 source/video/audio 数据被 Zod 深拷贝了两次。
  - 2.10 已为 `readLibrarySnapshot` 的 IPC handler 跳过了响应 schema parse，但 store 层的 `readSnapshot()` 内部仍有 `librarySnapshotDtoSchema.parse()`（L555），加上 lite 的二次 parse，总共三次对象图遍历。
  - 对 2842 容器规模，单次读取的峰值堆压力约 payload × 3~4。这是 CPU 3-4%（GC 受限）、磁盘 0（page cache 命中）、内存持续增长的直接原因。
- 实现建议（由低到高风险）
  - **F-1**：`readSnapshotLite()` 改为独立 SQL 查询，不经过 `readSnapshot()` 中转，消除一次完整 DTO 构造。
  - **F-2**：对大体积 DTO（snapshot/snapshotLite），在 store 层跳过 Zod `.parse()` 深拷贝，改为 `.safeParse()` 仅校验或直接 `as` 断言（与 2.10 在 IPC 层的 skip parse 对齐）。
  - **F-3**：评估 snapshot 读取的增量化/分页化——不一次性构造整个对象图，而是按需读取 source 子集。

#### 修正后的执行顺序

| 步骤 | 路线 | 内容 | 前置条件 |
|------|------|------|----------|
| 1 | A（修复版） | 修复 bench 状态泄漏 + 重跑交错复测 | 无 |
| 2 | F-1 + F-2 | `readSnapshotLite` 独立查询 + 跳过 store 层 Zod parse | A 通过 |
| 3 | B | 前端需求削峰（importBusy 时限制 resolve） | A 通过 |
| 4 | F-3 | 增量化快照（若 F-1/F-2 + B 仍不达标） | F-1/F-2 + B 完成 |
| 5 | E | Worker 迁移（仅在前述全部不足时） | 前述全部完成 |

- 路线 D（QoS）维持**跳过**。
- 路线 C 维持**降级为 B 的子项**。
- 路线 F 从 4.14 未覆盖的"建议方向"提升为**正式路线**，因为 8s 超时和内存特征已提供了充分的代码级证据。

### 4.21 Lite ON 根因修复（Bug 1 + Bug 2，2026-02-22）

#### 根因分析

**Bug 1：bench ready gate 在 Lite ON 下永远不满足**

- 调用链路：`mapImageSourceLiteDto` 将所有 package 映射为 `images: []` → `useImageBrowserViewModel` 中 `visibleImageRefs` 从 `activePackage.images.map(...)` 构建（空数组）→ `normalizeRefs`（`useEffectiveDisplayState.ts`）校验 `ref.imageIndex < pkg.images.length`，`images.length === 0` 时全部过滤 → `refsInPageCount === 0` → E2eBenchController ready gate（L529 `if (refsInPageCount <= 0)`）永远失败 → 永远卡在 `waiting_data`。
- 根本矛盾：Lite 快照的设计意图（不加载 `images[]` 降低内存）与 bench ready gate 的假设（`refsInPageCount > 0` 才算就绪）不兼容。
- 注：browsing 阶段的 data commit 由 `pageIndex === pending.toPage` 驱动，与 `refsInPageCount` 无关，Lite 模式下导航延迟测量不受影响；`imageTotalPagesEffective` 来自 `pageSnapshotForDisplay.totalItems`，与 Lite 无关，可正常计算。

**Bug 2：`invalidateCache()` 在活跃导入中触发 recovery 误杀运行任务（确定性 bug）**

- 调用链路：`executeImportTask`（`fileSystemImportTasks.ts:287`）调用 `params.invalidateCache()` → `stateHydrated = false` → 紧接着 `params.refreshSnapshot()` 内部调用 `ensureStateLoaded()` → 发现 `stateHydrated === false` → 运行 `recoverInterruptedImportTasks()` → 将当前正在运行的任务标记为 `failed` 并删除。
- 每次导入执行均确定性触发，即使最终导入成功（闭包 fallback 将 `completed` 写回），任务会被 recovery 短暂删除再重写，造成"双任务"现象和数据库额外写入。

#### 修复内容

- **`src/bench/E2eBenchController.tsx`**（Bug 1）
  - 组件顶层提取 `const isLiteSnapshot = benchSettings.librarySnapshotLite === true`。
  - ready gate 中将 `if (refsInPageCount <= 0)` 改为 `if (refsInPageCount <= 0 && !isLiteSnapshot)`，Lite ON 时跳过此门控。
  - `e2e_ready` 打点增加 `is_lite` 字段，deps 数组加入 `isLiteSnapshot`。

- **`electron/services/file-system-read/importTaskService.ts`**（Bug 2）
  - `recoverInterruptedImportTasks()` 中增加守卫：`if (task.status === 'running' && this.runningImportTaskIds.has(task.taskId)) continue`，跳过当前进程仍在运行的任务。
  - options 接口 `invalidateCache: () => void` 改为 `invalidateSnapshotCache: () => void`；`runImportTask()` 对应更新传参键名。

- **`electron/fileSystemReadFacade.ts`**（Bug 2）
  - 新增 `invalidateSnapshotCache()` 方法：`librarySnapshotService.invalidateCache()` + `importPathRegistry.hydrate(database.readImportSources())` + `mediaTokenService.cleanupExpiredTokens()`，**不重置 `stateHydrated`**，从而阻断 `ensureStateLoaded` → recovery 的调用链。
  - 关键设计：`importPathRegistry.hydrate()` 必须包含在内——快照扫描通过内存注册表（非直接读 DB）获取扫描路径，不重新水合则新写入的 import sources 不被识别，导致扫描结果为空（已通过测试回归验证）。

  - `ExecuteImportTaskParams` 接口 `invalidateCache` 改为 `invalidateSnapshotCache`；L288 调用点同步更新为 `params.invalidateSnapshotCache()`。

代码追踪确认了 `readSnapshotLite` 和 `readImageSidebarTree` 的数据源不一致：

调用链路：

```
readLibrarySnapshotLite:
  FileSystemLibraryHandlers.ts:40-44
  → ensureSnapshotLoaded()     // 仅触发加载，不使用返回值
  → database.readSnapshotLite() // 直接读 DB → 扫描期间返回空

readImageSidebarTree:
  libraryReadWriteServiceImpl.ts:247-273
  → ensureSnapshotLoaded()     // 使用返回的 snapshot 对象
  → filterLibrarySources(snapshot, request)  // 从内存 cache 构建树
```

时间线（Lite ON + 首次导入）：

```
T=0     preflight clearDatabase() → DB 清空、snapshotCache 清空
T=5s    warmup 完成 → enqueueImportTask
T=5s+   后端开始 loadSnapshot() 扫描
T=8s    persistCollectingPreviewSnapshot() → snapshotCache 有预览数据
        ↳ readImageSidebarTree → 返回有效树（sidebar labels 可用）
        ↳ readLibrarySnapshotLite → DB 仍空 → 返回空快照（无 packages）
        ↳ 前端：library.data.packages = [] → orderedRootScopedPackages = [] → sidebar 无内容可渲染
T=40s   replaceSnapshot() → DB 写入完整数据
        ↳ readLibrarySnapshotLite → 返回有效数据 → sidebar 渲染
```

**判定**：这是**程序设计缺陷**，不是测试设计需要。Lite 快照的设计意图是"薄读取 → 首屏立即可用 → 边导入边浏览"，但 `readLibrarySnapshotLite` 绕过了 `snapshotCache`，直接读 DB，而扫描期间 DB 为空。

**修复方案（A，最小改动）**：`readLibrarySnapshotLite` 在 DB 为空时回退到从 `snapshotCache` 构造 lite DTO。

- 文件：`electron/facade/FileSystemLibraryHandlers.ts` 的 `readLibrarySnapshotLite()`
- 改动：DB 读取结果为空（packages 数 = 0）时，从 `ensureSnapshotLoaded()` 返回的 snapshot 中提取 lite 数据
- 新增辅助函数 `buildLiteDtoFromSnapshot()` 将 full snapshot 映射为 lite DTO（剥离 `images[]`）

### 4.26 bench ready-gate 修复（insufficient_pages 立即退出，2026-02-22）

#### 问题发现

运行 Route B 验收 bench 时，所有 Lite ON 轮次均以 `reason=insufficient_pages` 立即退出（仅约 278ms）。

**根因**：4.23 实现的 `readLibrarySnapshotLite` snapshotCache fallback 在 preflight clear 之后仍保留了旧快照缓存，导致 `orderedPackages` 在 DB 清空后立即出现（从缓存取到包列表）。随后前端查询 DB 中的页数时返回 0-1（DB 已空），触发 `E2eBenchController.tsx` 中的：

```ts
// 旧代码（有问题）
if (!totalPages || totalPages <= 1) {
  readyGateStatsRef.current.total_pages_insufficient_count += 1
  void finish('insufficient_pages')  // 立即退出！
  return
}
```

**4.22 为何正常**：4.22 bench 运行时不存在 snapshotCache fallback，preflight clear 后快照返回空 → `orderedPackages=[]` → `preferredPackage=null` → bench 在 `preferred_missing_count` 循环等待约 40s 直到导入构建足够数据，从未触发 `insufficient_pages`。

#### 修复

`src/bench/E2eBenchController.tsx` L536-539：`total_pages <= 1` 时改为 `return`（继续等待）而非 `finish('insufficient_pages')`，由 `maxDurationMs` 超时兜底。

#### 影响

修复后：
- Lite ON 轮次：bench 等待约 40s 直到导入完成足够页数 → 与 4.22 行为一致
- `total_pages_insufficient_count` 计数器正常累积（r1=2，r2-r3=4），反映等待过程
- 旧的"仅 1 页时立即放弃"语义仅在 `requestNavigation` 中保留（浏览阶段合理失败）
