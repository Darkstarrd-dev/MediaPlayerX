## 二、核心问题诊断

### 问题 1：每个测试用例都重新 `render(<App />)` — 最大瓶颈

App 级测试文件中共有 **101 次** `render(<App />)` 调用。单独运行 `App.state.test.tsx` 的耗时分解：

```
transform:  3.64s   (TypeScript/JSX 编译)
import:     8.48s   (加载整个依赖图)
tests:     36.96s   (24 个测试各自渲染完整 App)
environment: 0.63s  (jsdom 初始化)
─────────────────
total:     46.30s
```

**每个** `it()` 块都执行一次 `render(<App />)`，意味着：
- 每次都重新实例化 `MockMediaRepository`（1179 行代码，含 6+ 处理器类和多个 Map）
- 每次都触发 `useAppController → useAppDataPipeline` 的完整 4 阶段数据管线
- 每次都构建完整的 React 组件树（AppShell → Sidebar + Main + Metadata + Fullscreen）

在 `App.state.test.tsx:54-60` 的 `beforeEach` 中还执行了全局状态重置：

```typescript
beforeEach(() => {
  vi.restoreAllMocks();
  resetUiStoreState();
  window.mediaPlayerBackend = undefined;
  window.localStorage.clear();
  window.sessionStorage.clear();
});
```

### 问题 2：大量 `waitFor()` 轮询等待 — 累积性能税

全部 App 测试文件中共有 **128 次** `waitFor()` 调用。`waitFor` 默认以 50ms 间隔轮询，最多等待 1000ms。很多用法只是在等待一次异步状态更新完成：

```typescript
// App.metadata.test.tsx:177 - 仅等一次 mock 被调用
await waitFor(() => {
  expect(searchExternalMetadata).toHaveBeenCalledTimes(2);
});
```

这种模式下，即使 mock 立即完成，`waitFor` 仍会有一次 50ms 的轮询周期延迟。128 次调用累积约 **6.4s** 的固定开销。

### 问题 3：`flushUiUpdates` 封装每个交互 — 微任务过度刷新

每个 App 测试文件都定义了相同的 helper 模式：

```typescript
const flushUiUpdates = async () => {
  await act(async () => { await Promise.resolve(); });
};
const click = async (target) => {
  fireEvent.click(target);
  await flushUiUpdates();
};
```

每次 `click`、`keyDown`、`mouseDown`、`wheel` 操作后都执行一次 `flushUiUpdates`。在一个测试中如果有 5-10 次交互操作，每次都触发完整的 React 调度周期。

### 问题 4：`import` 阶段耗时 8.5 秒 — 模块图过大

`App` 组件 import 了 `useAppController`，后者通过 `useAppDataPipeline` 引入 4 个阶段 hook，每个 hook 又 import 大量子模块。仅前端就有整个 `features/` 和 `components/` 的模块树。测试环境下没有 Vite 的 code splitting 优化，所以每个测试文件启动时都要解析整个依赖图。

### 问题 5：Vitest 配置缺少优化选项

当前 `vite.config.ts:68-83` 的 test 配置非常简单：

```typescript
test: {
  environment: "jsdom",
  setupFiles: "./src/test/setup.ts",
  globals: true,
  coverage: { ... },
}
```

缺少以下关键配置：
- `pool` / `poolOptions`：未配置线程池策略
- `fileParallelism`：文件间并行度未调优
- `testTimeout`：默认 5s，但 App 测试需要 25s，说明已预期慢
- `isolate`：未考虑是否降低隔离级别来共享模块缓存

### 问题 6：`App.fullscreen.test.tsx` 疑似存在 hang/超时循环

994 行代码、28 次 `waitFor`、20 次 `render(<App />)`，单独运行 5 分钟未完成。很可能其中某些 `waitFor` 条件在 jsdom 环境下永远无法满足（比如涉及到 `fullscreen API`、`ResizeObserver` 等浏览器特有行为），导致一直轮询到超时。

## 三、改善建议

### 优先级 1：修复 ThemeParameterPanel 循环依赖（预期：从挂死恢复正常）

**根因**：`ThemeParameterPanelMain.tsx` ↔ `ThemeParameterPanelPages.tsx` 存在循环导入。

```
ThemeParameterPanelMain.tsx
  → 导入 8 个组件 FROM ThemeParameterPanelPages.tsx
ThemeParameterPanelPages.tsx
  → 导入 ThemeParameterPageId, ThemeParameterPreviewMode FROM ThemeParameterPanelMain.tsx
```

这是最近 `refactor(theme): 拆分主题参数面板分页组件`（commit `28dee62`）引入的。拆分时两个类型（`ThemeParameterPageId` / `ThemeParameterPreviewMode`）留在了 Main 文件里，Pages 文件又反过来导入它们，形成了环。

**修复方式**：把 `ThemeParameterPageId` 和 `ThemeParameterPreviewMode` 提取到独立的类型文件（如 `themeParameterPanelTypes.ts`），打破循环。

### 优先级 2：减少 `render(<App />)` 次数（预期收益 50-60%）

**方案 A — 同 describe 内共享渲染实例**

对于同一个 describe 块内的多个 it，如果它们测试的是从同一个初始状态出发的不同操作，可以用 `beforeAll` 渲染一次，每个 it 只做断言：

```typescript
describe("元数据管理", () => {
  let container: RenderResult;
  beforeAll(async () => {
    container = render(<App />);
    await flushUiUpdates();
  });
  afterAll(() => container.unmount());
  // it(...) 只做操作 + 断言
});
```

**方案 B — 组件级测试替代 App 级测试**

将大部分测试下沉到具体组件级别，不再渲染整个 App。例如测试 "元数据评分" 时，直接 render `<MetadataPanel>` 并注入最小化的 props/context，而不是整个 App 树。

### 优先级 3：优化 `waitFor` 使用（预期收益 10-15%）

- 用 `findByRole`/`findByText`（内建 `waitFor`）替换 `waitFor(() => expect(screen.getByXxx(...)))`
- 对同步可验证的情况（mock 回调在 `act` 内已触发），直接用 `expect` 而非 `waitFor`
- 对必须用 `waitFor` 的场景，缩短超时时间：`waitFor(fn, { timeout: 200 })`

### 优先级 4：Vitest 配置优化（预期收益 15-20%）

```typescript
test: {
  environment: "jsdom",
  setupFiles: "./src/test/setup.ts",
  globals: true,
  pool: 'forks',           // 使用 fork 模式，比 threads 稳定
  poolOptions: {
    forks: { maxForks: 4 }, // 根据 CPU 核心数调整
  },
  testTimeout: 10_000,      // 全局超时降低
  hookTimeout: 10_000,
}
```

### 优先级 5：修复 `App.fullscreen.test.tsx` 的挂起问题

需要排查该文件中的哪些 `waitFor` 或浏览器 API（fullscreen、resize）在 jsdom 中不可用导致无限等待，加入必要的 mock 或降低 timeout。

### 优先级 6：拆分 flushUiUpdates 重复定义

所有 10 个 `App.*.test.tsx` 文件都重复定义了相同的 `flushUiUpdates`、`click`、`keyDown` 等 helper。提取到 `src/test/appTestUtils.ts` 中统一维护，降低维护成本（虽不直接影响性能）。

## 四、收益评估

| 改善项 | 涉及文件 | 预期收益 |
|--------|---------|---------|
| 修复 ThemeParameterPanel 循环依赖 | 2 个 theme-parameter 文件 | 从挂死恢复到 ~3s |
| 共享 App 渲染实例或下沉组件测试 | 10 个 App.*.test.tsx | 测试时间减少 50-60% |
| 优化 waitFor 使用 | 所有 128 处调用 | 减少 10-15% |
| Vitest 并行化配置 | vite.config.ts | 减少 15-20%（墙钟时间） |
| 修复 fullscreen hang | App.fullscreen.test.tsx | 从 >300s 降到 ~30s |

综合改善后，预期全套测试可从当前的 **>10 分钟**（含 fullscreen/ThemeParameterPanel hang）降至 **2-3 分钟**。
